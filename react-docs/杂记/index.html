JSX规则 ​

只返回一个根元素 ​

如果想要在一个组件中包含多个元素，需要用一个父标签把它们进行包裹，例如使用一个div标签，或者是一个Fragment标签:

js
<!-- div标签包裹 -->
<div>
  <h1>海蒂·拉玛的待办事项</h1>
  <img 
    src="https://i.imgur.com/yXOvdOSs.jpg" 
    alt="Hedy Lamarr" 
    class="photo"
    >
  <ul>
    ...
  </ul>
</div>

<!-- Fragment标签包裹 -->
<>
  <h1>海蒂·拉玛的待办事项</h1>
  <img 
    src="https://i.imgur.com/yXOvdOSs.jpg" 
    alt="Hedy Lamarr" 
    class="photo"
  >
  <ul>
    ...
  </ul>
</>

这是因为JSX看起来很像HTML，但在底层其实被转换为了多个 JavaScript 对象，由于不能在函数中返回多个对象，除非用一个数组把它们包裹起来

驼峰命令属性 ​

JSX 最终会被转化为 JavaScript，而 JSX 中的属性也会变成 JavaScript 对象中的键值对。在组件中，经常会遇到需要用变量的方式读取这些属性的时候。但 JavaScript 对变量的命名有限制。例如，变量名称不能包含 - 符号或者像 class 这样的保留字。

这就是为什么在 React 中，大部分 HTML 和 SVG 属性都用驼峰式命名法表示。例如，需要用 strokeWidth 代替 stroke-width。由于 class 是一个保留字，所以在 React 中需要用 className 来代替：

html
<img 
  src="https://i.imgur.com/yXOvdOSs.jpg" 
  alt="Hedy Lamarr" 
  className="photo"
/>

作为插槽传递 ​

当将一些自定义内容签嵌套在 JSX 标签中时，父组件将在名为 children 的 prop 中接受接受到对应内容，例如：

jsx
import Avatar from './Avatar.js';

  return (
    <div className="card">
      {props.children}
    </div>

}

export default function Profile() {

      <Avatar
        size={100}

        }}
        />
    </Card>

}

React中的Key ​

在列表渲染中，给列表代上Key，React 可以把它们匹配起来。这在数组项进行移动（例如排序）、插入或删除等操作时非常重要。一个合适的 key 可以帮助 React 推断发生了什么，从而得以正确地更新 DOM 树。

  • Key值在兄弟节点之间必须是唯一的
  • Key值不能随意改变 key可以让React从众多的兄弟元素中唯一标识出某一项（JSX 节点或文件）。而一个精心选择的 key 值所能提供的信息远远不止于这个元素在数组中的位置。即使元素的位置在渲染的过程中发生了改变，它提供的 key 值也能让 React 在整个生命周期中一直认得它，从而有利于复用。

TIP

不要在运行过程中动态地产生 key，像是 key={Math.random()} 这种方式。这会导致每次重新渲染后的 key 值都不一样，从而使得所有的组件和 DOM 元素每次都要重新创建。这不仅会造成运行变慢的问题，更有可能导致用户输入的丢失。

事件处理 ​

添加事件 ​

在React中必须传递给事件处理函数的是函数，而不是一个函数调用，虽然在Vue中是可行的，但在React中是万万不行的。 |------|------| |传递一个函数✔|传递一个函数调用❌| |<button onClick={handle}>|<button onClick={handle()}>|

第二个示例中，handleClick()会在 渲染 过程中立即触发函数，即使用户并没有去真的进行点击， 所以如果要在函数中传递参数，最好通过一个匿名函数进行包裹：

JSX
<button onClick={()=>handle('参数')}>按钮</button>

State ​

数据更新与响应 ​

jsx
function Parent(props) {

  let value = 1

   function parentHandleClick(){
    value = value + 1
    console.log('点击')
   } 

    return (
      <div onClick={parentHandleClick}>
        <p>Parent 组件 {value}</p>
        <div>
            <Children></Children>
        </div>
      </div>
    )
  }

在这个例子中handleParentClick事件处理函数通过不断点击去更新局部变量index。但是页面却不具有响应性，这是因为：

  • 局部变量无法在多次渲染中持久保存。当React再次渲染这个组件时，会从头开始渲染，而不会去考虑之前对局部变量的任何更改
  • 更改局部变量不会触发渲染。React 并不会意识到需要使用新数据来再次渲染组件。 对此，React会提供两个功能：
  • State 变量，用于保存渲染间的数据
  • State Setter 函数，更新变量并且触发 React 再次渲染组件
jsx
import { useState } from "react"

function Parent(props) {

  let [value,setValue] = useState(1)

   function parentHandleClick(){
    setValue(value + 1)
    console.log('点击')
   } 

    return (
      <div onClick={parentHandleClick}>
        <p>Parent 组件 {value}</p>
        <div>
            <Children></Children>
        </div>
      </div>
    )
  }

TIP

在React中，useState以及任何其他以“use”开头的函数都被称之为 Hook。 Hook 是特殊的函数，只在 React 的渲染时有效

在const [index, setIndex] = useState(0)中实际发生的行为如下：

  • 组件第一次渲染，将0作为index的初始值传递给useState，它将返回[0, setIndex]。React 记住了 0 是最新的 state 值。
  • 更新了state，会调用setIndex(index+1)。等于告诉了React现在的index是1，并且触发下一次渲染。
  • 组件进行第二次渲染。React 依然看到useState(0)，但是因为React记住了index被设置为1，所以返回[1, setIndex]

TIP

如果在一个组件中，使用多个useState，React 又是如何直到返回哪个 state 呢？

答案：

为了使语法更简洁，在同一组件的每次渲染中，Hooks 都依托于一个稳定的调用顺序。 在 React 内部，为每个组件保存了一个数组，其中每一项都是一个 state 对。它维护当前 state 对的索引值，在渲染之前将其设置为 “0”。每次调用 useState 时，React 都会为你提供一个 state 对并增加索引值。

另外 State 是屏幕上组件实例内部的状态。如果渲染同一个组件两次，每个副本都会有完全隔离的state。

State 完全私有于声明它的组件，父组件无法对其进行更改，如果希望多个相同组件同步一个State，最好的做法就是将State加入到共享的父组件中去。

渲染与提交 ​

组件显示在屏幕之前，必须被 React 渲染，渲染步骤是以下三大步：

  • 触发一次渲染
  • 渲染组件
  • 提交到DOM

触发第一次渲染 ​

有两种原因会导致组件的渲染：

  • 组件的 初次渲染，框架和沙箱有时会隐藏这部分代码，但它通过调用目标的 DOM 节点的createRoot，然后用组件调用render 。
  • 组件的 状态发生了变化，一般是使用set函数更新其状态之后触发的渲染，更新组件的状态会自动将一次渲染送入队列

渲染组件 ​

“渲染”即React在调用定义好的组件

  • 进行初次渲染时，React 会调用根组件
  • 对于后续的渲染，React 会调用内部状态更新触发渲染的函数组件

可知整个过程其实是递归的：如果更新后组件会返回某个另外的组件，那么React 接下来就会渲染 那组件，如果那组件又返回了某个组件，React 接下来会渲染这个“又”组件

TIP

渲染，必须是一次纯计算！！！，为了保证这种“纯”度，在React的严格模式下开发时，React会调用每个组件两次，从而帮助发现由副作用函数引起的错误

提交到DOM ​

在渲染组件之后，React会把“结果”提交到DOM上

  • 初次渲染，React会使用appendChild() DOM API 将其创建的所有 DOM 节点放在屏幕上
  • 对于重渲染，React将应用最少的必要操作（在渲染时计算），以使得DOM与最新的渲染输出相匹配

React 仅仅在渲染之间存在差异时才会去更改 DOM 节点

快照 ​

State 在它的表现出来的特性上来看，非常像“快照”这一概念

当调用set函数更新数据状态时，会立马进行一次“渲染”，在React中，渲染意味着 React 正在调用组件（本质上也就是一个函数）。从这个函数返回的 JSX 就像是UI的一张及时快照。它的props，事件处理函数和内部变量都是 根据当前渲染时的 state 所计算出来的。然后React 会更新屏幕来匹配这张快照。并且进行事件的绑定来处理函数。所以当React想要重新渲染一个组件时：

  • React 会再次调用组件函数
  • 函数返回新的 JSX 快照
  • React 更新界面来匹配这个快照

而State作为一个组件的记忆，State 不同于在函数中返回就会消失的局部变量。state 实际“活”在 React 本身中 —— 位于组件函数之外。 当React 调用组件函数时，它会为特定的那一次渲染提供一张 state 快照

来个经典例子：

jsx
import { useState } from 'react';

export default function Counter() {
  const [number, setNumber] = useState(0);

  return (
    <>
      <h1>{number}</h1>
      <button onClick={() => {
        setNumber(number + 1);
        setNumber(number + 1);
        setNumber(number + 1);
      }}>+3</button>
    </>
  )
}

在这段代码中，每次点击只会让number递增1，可以这么理解在当前的快照中，number的初始值始终为0

所以实际上的代码执行效果其实是这样:

jsx
<button onClick={() => {
  setNumber(number + 1);
  setNumber(number + 1);
  setNumber(number + 1);
}}>+3</button>

一个state变量的值永远不会在一次渲染的内部发生变化，即使其事件处理函数的代码是异步的。React 会使 state 的值始终“固定”在一次渲染的各个事件处理函数内部，这样就可以避免代码运行时 state 发生变化后引起一些不必要的麻烦。

批处理 ​

React 会等到事件处理函数中的 所有 代码都运行完毕再处理用户的state更新

这就是为什么重新渲染通常都发生再各种set函数调用之后

TIP

这样做的一个好处在于：当更新多个 state 变量时，不会触发太多的“重新渲染”。它不仅能帮助React应用运行得更快，它还会帮助避免处理只更新了一部分 state 变量的“半成品”渲染

但是如果实在是，想在下一次渲染之前更新同一个 state，就可以像setNumber(n => n + 1)

这其实是在告诉 React “用 state 值来做某事”而不是仅仅替换它的方法。

jsx
      <button onClick={() => {
        setNumber(n => n + 1);
        setNumber(n => n + 1);
        setNumber(n => n + 1);
      }}>+3</button>

对于更新函数来说，React会将其放入到一个队列中去更新，并且依靠上次的结果来迭代后面的更新函数的结果。

一句话总结：传入值等于是直接替代，传入更新函数需要依赖上次的结果

更新对象和数组 ​

虽然React中存放的对象是可变的，但实际上应该像处理数字和布尔值，字符串这样处理它们，替换整个对象，而不是对它们进行修改。

换句话说，应该把所有存放在 state 中的 JavaScript 对象都视为是只读的。

为了触发一次真正的渲染，应该创建一个新对象传入到 state 中，而不是直接去修改对象的属性。

js
// 正确做法
setPosition({
  x：e.clientX,
  y: e.clientY
})

// 错误做法
postion.x = e.clientX
postion.y = e.clientY

这里推荐使用 Immer 去更新对象

不要镜像props ​

非特殊情况下千万不要去镜像props：

js
function Message({messageColor}){
  const [color, setColor] = useState(messageColor)
}

此处，一个colorstate变量被初始化为messageColor的 prop 值。如果后面父组件传递不同的messageColor值，（例如，将blue改为red），会发现color不发生变化，state仅在第一次渲染期间进行了初始化。这和useState的底层实现有关。

但是如果，明确了需要忽略 特定props属性的所有更新时，将 props “镜像”到 state 时就具有意义了。按照常例， prop名称应该以initial或default开头，表示这个 prop 的新值就是用来忽略的：

js
function Message({ initialColor }) {
  // 这个 `color` state 变量用于保存 `initialColor` 的 **初始值**。
  // 对于 `initialColor` 属性的进一步更改将被忽略。
  const [color, setColor] = useState(initialColor);
  //。。。
}

保留与重置 ​

当在一个组件添加状态时，这个“状态”并不存在于组件内部。而是由React保存，React 通过组件在渲染树中的位置将它保存的每个状态与正确的组件关联起来。

当移除一个组件时，也会销毁它的state。

相同位置的相同组件会使得state被保留下来，相同位置的不同组件会让state重置。!!!! 这里说的“位置”，对React来说是组件在UI树的位置，而不是在JSX中的位置。

INFO

为什么说组件函数的定义不能被嵌套起来？？？

如果A组件嵌套了一个B组件，那么每次A组件渲染时，都会创建一个不同的“B”函数，等于是重新渲染一个B组件，这等于是在相同组件下渲染了“不同”的组件，所以React会将所有的state给重置掉，这会产生一些bug。

如果有时候，确实想在相同UI位置的组件不保留状态，可以为组件添加key(推荐做法)，或者改变组件位置（不推荐）

状态管理 ​

Reducer ​

有时候一个组件会拥有太多的状态更新逻辑，对于这种情况，最理想的做法是将组件的所有状态更新逻辑 都整合到一个外部函数里去，这个外部函数就叫做 Reducer

迁移useState ​

Reducer 是处理状态的另外一种方式，基本上依靠这三个步骤：

  • 1.将设置状态的逻辑修改成 dispatch 的一个 action 比如，一个修改列表数据源tasks的函数
js
function handleAddTask(text) {
  setTasks([
    ...tasks,
    {
      id: nextId++,
      text: text,
      done: false,
    },
  ]);
}

// 将handleAddTask

function handleAddTask(text){
  dispatch(
    // 传入 dispatch 的对象叫做“action”
    {
    type:'added',
    id: nextId++,
    text:text
  })
}

使用 Reducer 管理状态与直接设置状态的理念是不同的，直接设置状态来告诉React“要做什么”，而Recuder的理念是通过事件处理程序 dispatch 一个 “action”来表示“用户做了什么”。

INFO

action对象可以拥有多种结构，潜规则是添加一个字符串类型的type字段来描述发生了什么事情

  • 2.编写reducer函数 reducer函数就是放置状态逻辑的地方，它接受两个参数，分别是当前 state 和 action 对象，并且返回更新之后的state：
js
function yourReducer(state, action) {
  // 给 React 返回更新后的状态
}

React 会把状态设置为从 reducer 返回的状态。

要将状态设置逻辑从事件处理程序移到 reducer 函数中，需要：声明当前状态作为第一个参数；声明 action 对象作为第二个参数；从reducer返回下一个状态，React设置这个状态为新的状态。

js
function tasksReducer(tasks, action) {
  switch (action.type) 
  {
    case 'added': {
      return [
        ...tasks,
        {
          id: action.id,
          text: action.text,
          done: false,
        },
      ];
    }
    case 'changed':
    {
        //...
    }
    case 'deleted': 
    {
        //...
    }
    default: {
      throw Error('未知 action: ' + action.type);
    }
  }
}

TIP

之所以被称为 Reducer 也是有原因的：

因为其设计理念与数组中的reduce实际上是相同的。React中的reducer支持接受当前状态和action，然后返回下一个状态，这样action会随着事件推移累积到状态中

  • 3.在组件中使用reducer 先导入 reducer
js
import { useReducer } from 'react';

替换之前的 useState

js
// 这个useState可以被优化了
const [tasks, setTasks] = useState(initialTasks)
// 替换成 reducer
const [tasks,dispatch] = useReducer(taskReducer,initialTasks)

这样可以把大量的操作数据逻辑抽离到一个单独的文件中，现在，事件处理程序只需要通过派发action来指定发生了什么，而reducer函数通过响应actions来决定 状态如何更新

state和reducer的区别 ​

useState和useReducer在使用场景上存在一些区别：

  • 代码体积：通常，在使用 useState 时，一开始只需要编写少量代码。而 useReducer 必须提前编写 reducer 函数和需要调度的 actions。但是，当多个事件处理程序以相似的方式修改 state 时，useReducer 可以减少代码量
  • 可读性：当状态更新逻辑足够简单时，useState 的可读性还行。但是，一旦逻辑变得复杂起来，它们会使组件变得臃肿且难以阅读。在这种情况下，useReducer 允许你将状态更新逻辑与事件处理程序分离开来
  • 可调试性：当使用 useState 出现问题时, 你很难发现具体原因以及为什么。 而使用 useReducer 时， 你可以在 reducer 函数中通过打印日志的方式来观察每个状态的更新，以及为什么要更新（来自哪个 action）。 如果所有 action 都没问题，你就知道问题出在了 reducer 本身的逻辑中。 然而，与使用 useState 相比，你必须单步执行更多的代码
  • 可预测性：reducer 是一个不依赖于组件的纯函数。这就意味着你可以单独对它进行测试。一般来说，我们最好是在真实环境中测试组件，但对于复杂的状态更新逻辑，针对特定的初始状态和 action，断言 reducer 返回的特定状态会很有帮助
  • 个人偏好：并不是所有人都喜欢用 reducer，没关系，这是个人偏好问题。你可以随时在 useState 和 useReducer 之间切换，它们能做的事情是一样的

Context ​

基本使用 ​

context 可以避免数据多层的通过props传递到下层组件，步骤分为以下三步：

  • 创建一个context
js
// LevelContext.js
import { createContext } from 'react';

export const LevelContext = createContext(1);
  • 在需要数据的组件内部去 使用 这个创建出来的Context
js
import { useContext } from 'react';
import { LevelContext } from './LevelContext.js';

export default function Heading({ children }) {
  const level = useContext(LevelContext);
  // ...
}

useContext是一个Hook，useContext 告诉React Heading 组件想要读取 LevelContext

  • 在指定数据的组件中提供这个Context 在这里要使用它们，需要使用<Context.Provider>来进行包裹，以便于<LevelContext>来进行提供
jsx
import { LevelContext } from './LevelContext.js';

export default function Section({ level, children }) {
  return (
    <section className="section">
      <LevelContext.Provider value={level}>
        {children}
      </LevelContext.Provider>
    </section>
  );
}

这种做法其实就是在告诉React：如果在<Section>组件中的任何子组件请求LevelContext，给它们这个level。组件会使用UI树中它上层的最近的那个<LevelContext.Provider>`传递过来的值:

jsx
export default function Page() {
  return (
    <Section level={1}>
      <Heading>主标题</Heading>
      <Section level={2}>
        <Heading>副标题</Heading>
        <Heading>副标题</Heading>
        <Heading>副标题</Heading>
        <Section level={3}>
          <Heading>子标题</Heading>
          <Heading>子标题</Heading>
          <Heading>子标题</Heading>
          <Section level={4}>
            <Heading>子子标题</Heading>
            <Heading>子子标题</Heading>
            <Heading>子子标题</Heading>
          </Section>
        </Section>
      </Section>
    </Section>
  );
}

当然这个也有更加优雅的写法：

由于 context 可以让当前组件读取上层组件的信息，所以每个Section都可以从上层的Section读取Level

jsx
import { useContext } from 'react';
import { LevelContext } from './LevelContext.js';

export default function Section({ children }) {
  const level = useContext(LevelContext);
  return (
    <section className="section">
      <LevelContext.Provider value={level + 1}>
        {children}
      </LevelContext.Provider>
    </section>
  );
}

INFO

Context 让你可以编写“适应周围环境”的组件，并且根据在哪个context中来渲染它们不同的样子

Context 的工作方式可能会想起 CSS 属性继承。在 CSS 中，可以为一个 <div> 手动指定 color: blue，并且其中的任何 DOM 节点，无论多深，都会继承那个颜色，除非中间的其他 DOM 节点用 color: green 来覆盖它。类似地，在 React 中，覆盖来自上层的某些 context 的唯一方法是将子组件包裹到一个提供不同值的 context provider 中。

context的注意事项 ​

使用 Context 看起来非常好用！然而，这也意味着它也太容易被过度使用了。如果只想把一些 props 传递到多个层级中，这并不意味着你需要把这些信息放到 context 里

在使用 Context 之前，最好还是先考虑几种替代方案：

  • 坚持使用Props传递数据流
  • 抽象组件，并且将JSX作为children传递给它们。 如果你通过很多层不使用该数据的中间组件（并且只会向下传递）来传递数据，这通常意味着你在此过程中忘记了抽象组件。举个例子，你可能想传递一些像 posts 的数据 props 到不会直接使用这个参数的组件，类似 <Layout posts={posts} />。取而代之的是，让 Layout 把 children 当做一个参数，然后渲染 <Layout><Posts posts={posts} /></Layout>。这样就减少了定义数据的组件和使用数据的组件之间的层级

脱围机制 ​

ref ​

ref与state的区别 ​

refstate
useRef(value)返回的是{current:value}useState(value)返回的是[value,setValue]
更改时不会触发渲染更改时触发重新渲染
可变--在渲染过程之外修改和更新current的值“不可变”-- 必须使用state设置函数来修改state，从而排队重新渲染current的值
不应该在渲染期间读取current的值随时可以读取state。但是每次渲染都有自己不变的state 快照

ref的最佳实践 ​

  • 将 ref 视为脱围机制。当使用外部系统或者浏览器API时，ref很有用。比如存储 timeoutID，存储和操作 DOM 元素或者是其它不需要被用来计算JSX的其它对象

  • 不在渲染过程中读取和写入。ref本身就是一个普通的 JavaScript 对象，不参与React的渲染机制。

TIP

那为什么不直接使用一个普通局部变量来保存这些不需要渲染的变量？

实际上是不行的，如果某个state变量引起了组件的重新渲染，那么组件函数内部的所有局部变量都会被初始化，导致上一次渲染时存储的变量在这次渲染时被丢失了。

ref操作Dom ​

jsx
// 引入
import { useRef } from 'react';

// 声明
const myRef = useRef(null);

// 获取Dom
<div ref={myRef}>
  
// 在任意浏览器中可以使用APi了
myRef.current.scrollIntoView();

如果要获取批量的dom节点，比如一堆li，那么ref实际上可以接收一个函数来进行存储

jsx
<li
  key={cat.id}
  ref={node => {
    const map = getMap();
    if (node) {
      // Add to the Map
      map.set(cat, node);
    } else {
      // Remove from the Map
      map.delete(cat);
    }
  }}
>

同时需要注意的是，React不允许组件去访问其它组件的DOM节点，就算是自己的子组件也不行！ 这是特意为之

jsx
import { useRef } from 'react';

function MyInput(props) {
  return <input {...props} />;
}

export default function MyForm() {
  const inputRef = useRef(null);

  function handleClick() {
    //最终 inputRef 为 null
    inputRef.current.focus();
  }

  return (
    <>
      <MyInput ref={inputRef} />
      <button onClick={handleClick}>
        聚焦输入框
      </button>
    </>
  );
}

React认为手动操作 另外一个 组件的Dom节点是十分危险的行为。想要暴露其DOM节点必须确认好这个行为。那么可以使用forwardRefAPI：

jsx
const MyInput = forwardRef((props, ref) => {
  return <input {...props} ref={ref} />;
});

当然有时候，可以选择只暴露一部分功能给其它组件，那么可以使用useImperativeHandle这个方法:

jsx
const MyInput = forwardRef((props, ref) => {
  const realInputRef = useRef(null);
  useImperativeHandle(ref, () => ({
    // 只暴露 focus，没有别的
    focus() {
      realInputRef.current.focus();
    },
  }));
  return <input {...props} ref={realInputRef} />;
});

Effect ​

Effect 与 组件有不同的生命周期，组件可以挂载，更新或卸载。而Effect 只能做两件事情：开始同步某些东西，然后停止同步它。

Effect发生时机 ​

在理解Effect之前，需要了解两类逻辑类型：

  • 渲染逻辑代码: 位于组件的顶层。在这里接收 props 和 state，并对它们进行转换，最终返回在屏幕上看到的 JSX。渲染的代码必须是纯粹的——就像数学公式一样，它只应该“计算”结果，而不做其他任何事情。
  • 实践处理程序:嵌套在组件内部的函数，而不仅仅是计算函数。事件处理程序可能会更新输入字段、提交 HTTP POST 请求以购买产品，或者将用户导航到另一个屏幕。事件处理程序包含由特定用户操作（例如按钮点击或键入）引起的“副作用”（它们改变了程序的状态）

Effect往往发生在，不在渲染过程中，并且也没有特定的事件处理触发的时机下。

Effect 运行指定由渲染本身，而不是特定事件引起的副作用

声明Effect ​

声明Effect之后，可以在组件顶部对其进行调用:

jsx
import { useEffect } from 'react';

function MyComponent() {
  useEffect(() => {
    // 每次渲染后都会执行此处的代码
      if (isPlaying) {
        // 这里的ref是video元素
        ref.current.play();
      } else {
        ref.current.pause();
      }
  });
  return <div />;
}

useEffect会把这段代码放在屏幕更新渲染之后去执行，Effect 包裹着副作用，把副作用分离到渲染逻辑的过程之外。（上面的例子中，操作外界的媒体系统就是一种副作用）

TIP

一般来说，Effect 会在 每次 渲染之后执行，但是也有可能产生死循环:

比如下面的例子，每次渲染结束都会执行Effect；而更新state又会触发重新渲染....

jsx
const [count, setCount] = useState(0);
useEffect(() => {
  setCount(count + 1);
});

指定 Effect 依赖 ​

有时候不希望每次渲染的时候都去执行 Effect，而是希望特定的一些state变化之后再执行

那么可以将依赖数组传入useEffect的第二个参数，用来告诉React去跳过不必要地重新运行Effect

js
  useEffect(() => {
    if (isPlaying) { // isPlaying 在此处使用……
      // ...
    } else {
      // ...
    }
  }, [isPlaying]); // ……所以它必须在此处声明！

依赖数组可以包含多个依赖项。当指定的所有依赖项在上一次渲染期间的值与当前值完全相同时，React 会跳过重新运行该 Effect。

另外，useEffect的第二个参数也有各种情况：

jsx
useEffect(() => {
  // 这里的代码会在每次渲染后执行
});

useEffect(() => {
  // 这里的代码只会在组件挂载后执行
}, []);

useEffect(() => {
  //这里的代码只会在每次渲染后，并且 a 或 b 的值与上次渲染不一致时执行
}, [a, b]);

添加cleanup函数 ​

这里需要注意一个点：在开发环境中，React会挂载组件两次。

这是React特意为之，避免一些细节上的错误不被发现：比如是否及时清理一些遗留的订阅事件或者网络连接。

jsx
  useEffect(() => {
    const connection = createConnection();
    connection.connect();
    // cleanup函数
    return () => {
      connection.disconnect();
    };
  }, []);

“两次”挂载 ​

对于React的特意重复挂载组件，正确保留的态度应该是“如何修复Effect以便于它在重复挂载之后仍然能够正确工作”

  • 正确关闭一些“对话框”，某些API不允许连续调用两次，比如一些组件库得到<dialog>
jsx
useEffect(() => {
  const dialog = dialogRef.current;
  dialog.showModal();
  // 卸载的时候记得清理
  return () => dialog.close();
}, []);
  • 正确清理一些订阅事件
jsx
useEffect(() => {
  function handleScroll(e) {
    console.log(window.scrollX, window.scrollY);
  }
  window.addEventListener('scroll', handleScroll);
  return () => window.removeEventListener('scroll', handleScroll);
}, []);
  • 正确重置一些动画
jsx
useEffect(() => {
  const node = ref.current;
  node.style.opacity = 1; // 触发动画
  return () => {
    node.style.opacity = 0; // 重置为初始值
  };
}, []);
  • 避免一些重复请求，或者添加一些请求缓存的响应方案
jsx
useEffect(() => {
  let ignore = false;

  async function startFetching() {
    const json = await fetchTodos(userId);
    if (!ignore) {
      setTodos(json);
    }
  }

  startFetching();

  return () => {
    ignore = true;
  };
}, [userId]);

因为无法撤销已经发生过的网络请求，所以在开发环境中看到两条相同的网络请求实际上是正确的

缓存昂贵的计算 ​

如果一个组件使用它接收到的 props 中的filter对另一个prop todos 进行筛选，计算得出visibleTodos。如果这个计算很麻烦，你并不想在每次渲染中都去计算它，这里更加推荐使用useMemo而不是useEffect

jsx
import { useMemo, useState } from 'react';

function TodoList({ todos, filter }) {
  const [newTodo, setNewTodo] = useState('');
  // ✅ 除非 todos 或 filter 发生变化，否则不会重新执行 getFilteredTodos()
  const visibleTodos = useMemo(() => getFilteredTodos(todos, filter), [todos, filter]);
  // ...
}

这种写法告诉了React，除非todos或filter发生了变化，否则不要去重新执行传入的函数。React 会在初次渲染的时候记住 getFilteredTodos() 的返回值。在下一次渲染中，它会检查 todos 或 filter 是否发生了变化。如果它们跟上次渲染时一样，useMemo 会直接返回它最后保存的结果。如果不一样，React 将再次调用传入的函数（并保存它的结果）。

TIP

useEffect与useMemo最大的区别在于： useMemo只适合与做纯函数计算的场景，而useEffect支持去做组件副作用相关的操作

受控组件 ​

受控组件的定义：满足state是表单组件的唯一数据源且控制表单操作并同步state这两个特点的组件并称之为受控组件。由于数据的来源是React的状态管理，所以认为是受控的。

jsx
// 以下就是一个受控组件的例子
import React from "react";


export default class Parent extends React.Component {
  constructor(props) {
    super(props);
  }

  // state 是组件的唯一数据源
  state = {
    username :''
  }

  // 控制表单的操作去同步state

  handleUserNameChange(e){
    this.setState({
      username: e.target.value
    })
  }

  render() {
    const {username} = this.state

    return (
      <form>
        <p>
          用户名：
          <input type="text" 
          placeholder="用户名" 
          value={username} 
          onChange={this.handleUserNameChange.bind(this)}/>
        </p>
      </form>
    )
  }
}

使用受控组件的好处是可以方便地获取和更新表单元素的值，以及在 React 组件中对表单数据进行处理和验证。另外，受控组件将数据和视图分开，受控组件也使得表单元素的值可以与其他组件状态或全局状态进行交互，从而实现更复杂的表单逻辑。

同时也存在 非受控组件 ：表单数据不受控于state，使用 React ref 从DOM节点中获取表单数据的组件

jsx
import React from "react";

export default class Parent extends React.Component {
  constructor(props) {
    super(props);
    this.usernameRef = React.createRef()
  }

  handleSubmitClick(e) {
    e.preventDefault()
    console.log(this.usernameRef.current.value)
  }

  render() {
    return (
      <>
        <form>
          <p>
            用户名：
            {/* defaultValue 可以设置默认值 */}
            <input type="text" 
              placeholder="用户名" 
              ref={this.usernameRef} 
              defaultValue="默认值"/>
          </p>
        </form>
        <button onClick={this.handleSubmitClick.bind(this)}>提交</button>
      </>
    );
  }
}

非受控组件的值不再受组件自身的state和props所控制，数据存在了DOM之中，获取依靠refs来进行读取，由于DOM引用不受React的控制，所以叫做非受控组件。

受控组件和非受控组件之间的使用场景可以这样来区分：在非常简单的业务表单中，可以选择非受控组件；如果表单趋于复杂，不得不选择受控组件，主要原因还是在于状态管理上的灵活性，受控组件也使得表单元素的值可以与其他组件状态或全局状态进行交互
